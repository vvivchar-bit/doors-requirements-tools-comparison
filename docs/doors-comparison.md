# Порівняльна характеристика IBM DOORS та DOORS Web Access

## 1. Вступ

У межах навчального проєкту «Програмна система “Готель”» важливо забезпечити правильне управління вимогами, їх структурування, трасування, контроль змін і можливість спільної роботи учасників проєкту.

Для цього можуть використовуватися спеціалізовані інструменти управління вимогами, зокрема IBM Engineering Requirements Management DOORS та DOORS Web Access.

IBM DOORS є інструментом для управління вимогами, який дозволяє створювати модулі вимог, описувати атрибути, встановлювати зв’язки між вимогами, аналізувати зміни та забезпечувати трасування на різних етапах життєвого циклу програмного забезпечення.

DOORS Web Access є веб-клієнтом, який забезпечує доступ до вимог через браузер і дає змогу залучати до роботи користувачів, які не використовують повний настільний клієнт DOORS.

---

## 2. Загальні відомості про IBM DOORS

IBM Engineering Requirements Management DOORS використовується для управління вимогами у складних програмних і системних проєктах.

Основні можливості DOORS:

- створення та редагування вимог;
- структурування вимог у модулях;
- налаштування атрибутів вимог;
- трасування зв’язків між вимогами;
- контроль змін;
- аналіз впливу змін;
- робота з великими наборами вимог;
- підтримка DXL для автоматизації окремих операцій.

Для проєкту «Програмна система “Готель”» такий інструмент може бути використаний для збереження функціональних і нефункціональних вимог, зв’язку вимог із тестами, контролю змін та перевірки повноти реалізації функцій.

---

## 3. DOORS Client для Windows

DOORS Client для Windows є основним повнофункціональним клієнтом для роботи з вимогами.

Він дає змогу:

- створювати модулі вимог;
- редагувати вимоги;
- створювати атрибути;
- виконувати трасування;
- аналізувати зміни;
- працювати з DXL;
- адмініструвати структуру вимог.

Переваги Windows-клієнта:

- найбільш повний функціонал;
- зручний для аналітика вимог;
- підтримує глибоку роботу з модулями;
- дозволяє виконувати трасування між вимогами;
- підходить для повного адміністрування вимог.

Обмеження:

- потребує встановлення клієнтського програмного забезпечення;
- орієнтований на Windows-середовище;
- складніший для користувачів-початківців;
- надлишковий для невеликого навчального проєкту.

Для проєкту «Готель» DOORS Client для Windows доцільно використовувати як основний інструмент аналітика, якщо потрібно повноцінно керувати вимогами, трасуванням і змінами.

---

## 4. DOORS для Linux

Linux доцільно розглядати як платформу для серверної інфраструктури, пов’язаної з DOORS або DOORS Web Access.

Переваги Linux-середовища:

- стабільність серверної роботи;
- можливість розгортання серверних компонентів;
- зручність для корпоративної інфраструктури;
- можливість централізованого доступу до вимог.

Обмеження:

- не є основним середовищем для повноцінного настільного клієнта DOORS у сучасній версії;
- потребує адміністрування сервера;
- складніше для навчального використання;
- не є необхідним для невеликого проєкту.

Для навчального проєкту «Готель» Linux доцільний лише як варіант серверної інфраструктури, якщо потрібно організувати доступ до вимог через DOORS Web Access.

---

## 5. DOORS для Solaris

Solaris можна розглядати як історичну платформу, яка використовувалась у деяких корпоративних середовищах.

Однак для актуального навчального проєкту використання Solaris недоцільне.

Причини:

- у сучасній версії DOORS 9.7.2.10 Solaris не підтримується;
- DOORS Web Access 9.7.2.10 також не підтримує Solaris;
- платформа є менш поширеною для навчальних і малих проєктів;
- налаштування середовища складніше, ніж для Windows або Linux.

Для проєкту «Програмна система “Готель”» Solaris розглядається лише як пункт порівняння, але не як практична платформа для використання.

---

## 6. DOORS Web Access

DOORS Web Access є веб-клієнтом, який забезпечує доступ до вимог через браузер.

Його основне призначення — надати можливість переглядати вимоги, брати участь у погодженні, коментуванні та спільній роботі без встановлення повного настільного клієнта DOORS.

Переваги DOORS Web Access:

- доступ через браузер;
- зручність для замовника, викладача або учасників команди;
- не потребує встановлення повного клієнта;
- підходить для перегляду вимог;
- спрощує спільну роботу;
- зручний для погодження змін.

Обмеження:

- не має всіх можливостей повного DOORS Client;
- залежить від серверної частини;
- потребує налаштування інфраструктури;
- більше підходить для перегляду та спільної роботи, ніж для повного адміністрування.

Для проєкту «Готель» DOORS Web Access є зручним інструментом для демонстрації вимог викладачу або замовнику, перегляду структури вимог і погодження змін.

## 8. Застосування до проєкту «Програмна система “Готель”»

У межах проєкту «Готель» IBM DOORS може використовуватися для управління такими групами вимог:

| Категорія вимог | Приклади вимог |
|---|---|
| Функціональні вимоги | Перегляд номерів, пошук номерів, створення бронювання, скасування бронювання |
| Нефункціональні вимоги | Зручність інтерфейсу, швидкість роботи, надійність збереження даних |
| Вимоги до даних | Ім’я клієнта, телефон, email, номер кімнати, дата заїзду, дата виїзду |
| Вимоги до інтерфейсу | Статистика готелю, список номерів, форма бронювання, список бронювань |
| Вимоги до тестування | Перевірка пошуку, бронювання, LocalStorage, відсутності помилок у консолі |

Приклад структурування вимог у DOORS:

| ID | Вимога | Тип | Пріоритет |
|---|---|---|---|
| REQ-01 | Система повинна відображати список номерів | Функціональна | Високий |
| REQ-02 | Система повинна дозволяти пошук номерів за типом | Функціональна | Високий |
| REQ-03 | Система повинна створювати бронювання | Функціональна | Високий |
| REQ-04 | Система повинна перевіряти обов’язкові поля форми | Функціональна | Високий |
| REQ-05 | Система повинна зберігати дані у LocalStorage | Технічна | Середній |
| REQ-06 | Інтерфейс системи має бути зрозумілим для користувача | Нефункціональна | Середній |

## 9. Переваги та недоліки для навчального проєкту

### Переваги використання DOORS

- централізоване зберігання вимог;
- контроль змін;
- трасування вимог;
- можливість пов’язувати вимоги з тестами;
- підтримка командної роботи;
- підходить для складних проєктів.

### Недоліки використання DOORS

- складність налаштування;
- надлишковість для невеликого навчального проєкту;
- потреба у спеціалізованих знаннях;
- залежність від ліцензування та інфраструктури;
- повноцінна робота потребує встановлення клієнта або серверних компонентів.

---

## 10. Висновок

У результаті порівняння було розглянуто основні варіанти використання IBM DOORS та DOORS Web Access для платформ Windows, Linux, Solaris і веб-доступу.

Для навчального проєкту «Програмна система “Готель”» найбільш доцільним є використання DOORS Client для Windows і DOORS Web Access.

DOORS Client для Windows підходить для повного створення, редагування, структурування і трасування вимог. DOORS Web Access доцільно використовувати для перегляду вимог, погодження змін і спільної роботи учасників проєкту.

Linux доцільно розглядати як серверну платформу для інфраструктури DOORS Web Access. Solaris у межах актуальної версії DOORS 9.7.2.10 не підтримується, тому для проєкту «Готель» не є доцільним.

Отже, для проєкту «Готель» оптимальним рішенням є поєднання повного клієнта DOORS для Windows із веб-доступом DOORS Web Access для спільної роботи та демонстрації вимог.
